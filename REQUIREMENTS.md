# 네온 성당 (Neon Cathedral) — 요구사항 명세서

> **VaporBurn (가칭)** — "흔적 없이 소멸하는 고해성사, 오직 따뜻한 불빛으로만 온기를 나누는 공간"

- 문서 버전: v0.1 (초안)
- 작성일: 2026-07-08
- 상태: 컨셉 확정, 요구사항 정의 단계

---

## 1. 프로젝트 개요

### 1.1 서비스 정의
익명으로 고민(고해)을 작성하고, 작성한 글을 '태워서' 영구 소멸시키는 감성 힐링 커뮤니티.
타인의 고민에는 '촛불'로만 공감할 수 있으며, 깊은 공감을 받은 글만 예외적으로 '스테인드글라스'로 박제된다.

### 1.2 핵심 가치
| 가치 | 설명 |
|---|---|
| **완벽한 익명** | 회원가입 없음. 개인정보를 일절 수집하지 않고 자동 부여 식별자만 사용 |
| **소멸의 카타르시스** | 글은 태우는 순간 영구히 사라짐 — 기록이 남지 않는 안전함 |
| **따뜻한 공감** | 댓글/텍스트 반응 없이 오직 '촛불 켜기'로만 온기를 전달 |
| **심리적 안정감** | 로파이 사운드 + 어두운 성당 분위기의 몰입형 공간 |

### 1.3 타겟 사용자
- 고민을 털어놓고 싶지만 신원 노출과 기록이 부담스러운 사용자
- 악플·판단 없는 안전한 감정 해소 공간을 원하는 사용자
- 감성적·몰입형 웹 경험을 선호하는 10~30대

## 2. 디자인 요구사항

**디자인 테마는 사전에 정의하지 않는다. 이 문서의 컨셉을 보고, 구현하는 Claude가 드리블(Dribbble)에 올라올 만한 느낌으로 재량껏 디자인한다.**

### 2.1 모바일 퍼스트 반응형 (Mobile-First Responsive)
- **플랫폼**: 반응형 웹 단일 코드베이스. 모바일 화면을 기준으로 먼저 디자인·구현하고, 태블릿/데스크톱으로 확장한다.
- **레이아웃 원칙**:
  - 모든 화면은 모바일 뷰포트(360px~)에서 완결된 경험을 제공해야 한다. 데스크톱은 여백·빛 연출 확장 등 "더 넓은 성당"으로 확장하는 개념.
  - 스테인드글라스 벽 등 그리드형 화면은 브레이크포인트별 열 수 조정 (모바일 2열 → 데스크톱 다열 모자이크).
- **터치 우선 인터랙션**:
  - 촛불 켜기 글로우 확산은 롱프레스(터치 홀드) 기준으로 설계하고, 데스크톱에서는 마우스 홀드로 동일하게 동작.
  - 터치 타겟 최소 44×44px, 하단 도달 가능 영역(thumb zone)에 핵심 액션 배치.
- **모바일 성능**: 연소 파티클·글로우 효과는 중급 모바일 기기에서 60fps 기준으로 최적화. 뷰포트 크기에 따라 파티클 수 조절.
- **모바일 브라우저 대응**: 주소창 높이 변화(`dvh` 단위), 사파리 세이프 에어리어(notch), 가로 모드 처리.

### 2.2 성능 기준
- [ ] 파티클/글로우 애니메이션 60fps 유지 (저사양 기기용 감소 모드 제공)
- [ ] `prefers-reduced-motion` 접근성 대응

## 3. 기능 요구사항 (Functional Requirements)

### FR-1. 비회원가입(No Sign-up) 익명 시스템
- **FR-1.1** 회원가입 절차 자체가 존재하지 않는다. 사용자는 아무 정보 입력 없이 접속 즉시 서비스를 이용할 수 있으며, 개인정보(이메일, 전화번호, 이름 등)를 일절 수집하지 않는다.
- **FR-1.2** 최초 방문 시 순번 기반 익명 식별자를 자동 부여한다. (예: "42번째 고해자")
- **FR-1.3** 기기 기반 세션 유지 (익명 토큰). 기기 변경 시 계정 복구 수단은 제공하지 않거나, 익명성을 해치지 않는 복구 코드 방식만 검토한다.
- **FR-1.4** 프로필/팔로우/DM 등 관계 형성 기능은 의도적으로 제공하지 않는다.

### FR-2. 고해(고민 글) 작성 & 소멸
- **FR-2.1** 사용자는 텍스트로 고민을 작성할 수 있다. (최대 글자 수 정의 필요, 예: 2,000자)
- **FR-2.2** 작성 완료 후 '태우기' 버튼을 누르면 연소 애니메이션과 함께 글이 화면에서 사라진다.
- **FR-2.3** 태워진 글은 공개 피드에 익명으로 노출된다. (노출 기간/소멸 조건은 §FR-3, §FR-4와 연동)
- **FR-2.4** 소멸된 글은 **서버에서도 영구 삭제**한다. 작성자 본인도 다시 볼 수 없다. (단, AI 답장 생성 및 스테인드글라스 박제 판정에 필요한 최소 보존 기간은 §7.2 참고)
- **FR-2.5** 수정/삭제 개념 없음 — 태우기 전에만 편집 가능.

### FR-3. 촛불 공감 시스템
- **FR-3.1** 다른 사용자의 글에 '촛불 켜기'로 공감할 수 있다. (글 1개당 사용자 1회)
- **FR-3.2** 텍스트 댓글·답글 기능은 제공하지 않는다. (악플 원천 차단)
- **FR-3.3** 촛불 개수는 글에 시각적으로 표시된다. (숫자 + 촛불 밝기/개수 시각화)
- **FR-3.4** 촛불 버튼을 길게 누르는 동안 화면 글로우 확산 인터랙션이 동작한다.
- **FR-3.5** 프리미엄 캔들(오로라, 홀로그램 등) 스킨 적용 가능. (§FR-6 수익화 연동)

### FR-4. 스테인드글라스 벽화 (박제 시스템)
- **FR-4.1** 촛불 **100개 이상**을 받은 글은 소멸하지 않고 스테인드글라스 조각으로 영구 박제된다.
- **FR-4.2** 박제 시점에 작성자에게 알림(익명 유지)을 제공한다. 작성자가 박제를 거부할 수 있는 옵트아웃 제공 여부 결정 필요. *(권장: 제공)*
- **FR-4.3** 스테인드글라스 벽 페이지에서 박제된 사연들을 미술관 관람하듯 탐색할 수 있다.
- **FR-4.4** 유리 조각의 색/형태는 글의 감정 톤 또는 촛불 수에 따라 달라진다. (선택 사항)
- **FR-4.5** 박제 임계값(100개)은 서버 설정으로 조정 가능해야 한다.

### FR-5. 고독한 참회 도우미 (답장 시스템)
> **기술 결정**: LLM API를 사용하지 않는다(§6.1). 답장은 사전 집필된 편지 라이브러리에서 톤·주제에 맞게 선별/조합하여 제공한다.

- **FR-5.1** 글 작성(태우기) 시 답장 톤 카드를 선택할 수 있다:
  - **악마의 속삭임** — 냉철한 이성, 직설적 조언
  - **천사의 위로** — 따뜻한 감성, 공감과 위로
- **FR-5.2** 태운 뒤 **5분 후** 선택한 톤의 답장이 '편지봉투' 형태로 본인에게만 도착한다.
- **FR-5.3** 답장 생성 방식: 톤별로 사전 집필된 편지 문구 풀(pool)에서 선별. 키워드 기반 주제 매칭 등 개인화 수준은 콘텐츠 제작 단계에서 결정한다.
- **FR-5.4** 편지 봉투(답장권)는 하루 1회 무료. 추가 답장은 유료(§FR-6).
- **FR-5.5** 자살·자해 등 위기 키워드 감지(규칙 기반) 시 전문 상담 기관 안내를 답장에 포함한다. (§7.4 안전 요구사항)

### FR-6. 수익화 (Monetization)
- **FR-6.1 편지 봉투**: 추가 답장권 패키지 판매. (예: 봉투 5개 2,900원)
- **FR-6.2 프리미엄 캔들**: 특수 촛불 스킨(오로라, 홀로그램 등) 판매.
- **FR-6.3** 결제는 익명성을 해치지 않는 방식으로 처리한다. (결제 정보와 익명 계정의 분리 저장)
- **FR-6.4** 구매 내역 조회 및 환불 정책 페이지 제공. (법적 필수)

### FR-7. 신고 & 모더레이션
- **FR-7.1** 공개 피드 글에 대한 신고 기능. (불법 정보, 타인 비방 등)
- **FR-7.2** 규칙·키워드 기반 유해 콘텐츠 사전 필터링. (개인정보 노출, 실명 언급, 불법 콘텐츠) — LLM 미사용 원칙(§6.1)에 따라 규칙 기반으로 구현
- **FR-7.3** 관리자용 모더레이션 도구. (신고 처리, 박제 글 관리)

## 5. 사용자 흐름 (Core User Flow)

```
입장 (익명 세션 발급: "N번째 고해자")
  │
  ▼
고해실에서 고민 작성
  │
  ▼
톤 카드 선택 (악마의 속삭임 / 천사의 위로)
  │
  ▼
"태우기" → 연소 파티클 애니메이션 → 글이 공개 피드로
  │
  ├──▶ 5분 후: 답장 편지봉투 도착 (본인에게만)
  │
  ▼
피드에서 촛불 공감 누적
  │
  ├── 촛불 < 100 & 노출 기간 만료 ──▶ 영구 소멸 (서버 삭제)
  │
  └── 촛불 ≥ 100 ──▶ 스테인드글라스 박제 (영구 보존)
```

## 6. 기술 스택 (확정)

### 6.1 확정 스택
| 영역 | 결정 | 비고 |
|---|---|---|
| 프레임워크 | **Next.js 최신 버전** (App Router) + TypeScript | Vercel 배포 |
| 스타일링 | **Tailwind CSS 최신 버전** | 디자인 토큰은 Tailwind 테마로 시스템화 |
| 백엔드 | **Server Actions 전용** | **API Route 사용 금지.** 외부 노출이 필요한 API가 생기는 경우에만 예외적으로 검토 |
| DB | **당분간 미사용** | 필요 시점에 **Supabase DB** 추가. 그 전까지는 서버 메모리/목업 데이터로 개발 |
| AI/LLM | **LLM API 미사용** | AI 답장 기능은 LLM 호출 없이 구현 (§FR-5 참고) |
| 애니메이션 | Canvas 파티클 (연소 효과), CSS + **Motion** (글로우·전환) | 구 Framer Motion, `motion` 패키지. 성능 임계 구간은 WebGL 검토 |
| 사운드 | Web Audio API | 로파이 앰비언트 루프 |
| 인증 | 익명 토큰 (쿠키 기반) | 회원가입 없음, 개인정보 미수집 |

### 6.2 추후 결정
| 영역 | 시점 | 비고 |
|---|---|---|
| DB 스키마 & TTL 삭제 잡 | Supabase 도입 시 | "영구 소멸" 정책(§7.2)에 맞는 하드 삭제 설계 |
| 결제 PG | Phase 3 (수익화) | 국내 PG (토스페이먼츠 등), 익명 계정과 분리 저장 |
| 답장 지연 발송 구현 | 답장 기능 구현 시 | 5분 딜레이 처리 방식 (클라이언트 타이머 vs 서버 스케줄) |

## 7. 비기능 요구사항 (Non-Functional)

### 7.1 프라이버시
- 개인정보 수집 항목: **없음** (결제 시 PG사 처리 정보 제외)
- IP 등 접속 로그는 법정 최소 기간만 보관하고 콘텐츠와 연결하지 않는다.
- LLM API를 사용하지 않으므로(§6.1) 고해 내용이 외부 AI 서비스로 전송되지 않는다 — 이를 프라이버시 강점으로 서비스 소개에 명시할 수 있다.

### 7.2 데이터 소멸 정책
- 소멸 대상 글은 노출 기간 종료 시 DB에서 **하드 삭제** (soft delete 금지).
- 단, 답장 발송 대기(최대 5분)와 박제 판정에 필요한 기간 동안만 임시 보존.
- 백업/로그에 잔존하는 데이터의 삭제 주기 정의 필요.

### 7.3 성능·가용성
- 피드 로딩 1.5초 이내 (P75), 애니메이션 60fps 목표.
- 답장 지연 발송의 정확성 (5분 ± 30초).

### 7.4 안전 (Trust & Safety)
- 위기 신호(자살·자해 암시) 감지 시 상담 기관 연락처 안내. (예: 자살예방상담전화 109)
- 미성년자 보호: 유해 콘텐츠 필터링 기준 강화.
- 청소년보호정책, 커뮤니티 가이드라인 페이지 필수.

### 7.5 접근성
- `prefers-reduced-motion` 대응 (애니메이션 감소 모드).
- 스크린리더 대응 (핵심 플로우 ARIA 처리).
- 명도 대비: 어두운 배경 위 텍스트 WCAG AA 이상.

## 9. MVP 범위 제안

**Phase 1 (MVP)**
- 익명 세션 + 고해 작성 + 연소 애니메이션(간소화 버전)
- 공개 피드 + 촛불 공감 (기본 촛불만)
- 답장 시스템 (하루 1회 무료, 톤 카드 2종, 사전 집필 편지 기반)
- 데이터 자동 소멸

**Phase 2**
- 스테인드글라스 벽화 (박제 시스템)
- 앰비언트 사운드, 글로우 인터랙션 고도화

**Phase 3**
- 결제 (편지 봉투 패키지, 프리미엄 캔들)
- 모더레이션 도구 고도화
